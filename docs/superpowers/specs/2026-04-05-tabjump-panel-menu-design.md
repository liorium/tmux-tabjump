# tmux-tabjump 패널형 메뉴 UX 설계

작성일: 2026-04-05  
상태: 승인됨 (사용자 승인 완료, 구현 전)

## 1. 요구사항 요약

사용자는 현재 `display-menu` 기반의 작은 계층 메뉴가 UI/UX 측면에서 답답하다고 느끼고 있다. 특히 가장 중요한 작업인 **현재 pane을 기존 탭에 붙이는 흐름**이 메뉴 안에서 너무 깊고, 큰 화면을 충분히 활용하지 못한다고 판단했다.

확정된 우선순위는 다음과 같다.

### 최우선 작업
1. 현재 pane을 기존 탭에 붙이기
2. 현재 pane을 새 탭에 붙이기

### 탭 관리 우선순위
1. 순서 변경
2. 이름 변경
3. 삭제
4. dead/empty 정리

### 탭 지속성 규칙
- 사용자가 만든 탭의 **이름 / 순서 / 존재 자체**는 항상 저장된다.
- 나중에 연결된 pane이 사라졌더라도 탭은 유지된다.
- 사라진 pane과 연결돼 있던 탭은 복원 시 **dead가 아니라 empty 상태**로 보인다.

---

## 2. 선택한 방향

세 가지 UX 방향 중에서 사용자는 **B. 2열 패널형**을 선택했다.

### 선택 이유
- 큰 화면을 자연스럽게 활용할 수 있다.
- **상태(탭 목록)** 와 **행동(붙이기/관리)** 를 동시에 보여줄 수 있다.
- 최우선 작업인 “현재 pane을 기존 탭에 붙이기”를 가장 짧은 흐름으로 만들 수 있다.
- 계층 메뉴보다 현재 작업 맥락을 더 명확하게 전달할 수 있다.

### 채택한 기본 진입 방식
패널을 열자마자 **기존 탭에 붙이기 모드**가 기본으로 열린다.

즉 기본 흐름은:
1. 메뉴 열기
2. 오른쪽 탭 리스트에서 대상 탭 선택
3. `Enter` 또는 클릭으로 attach

---

## 3. 비채택 대안

### A. 개선된 계층형 `display-menu`
**장점**
- 기존 코드 재사용 폭이 크다.
- 구현량이 비교적 적다.

**단점**
- 여전히 단계가 깊다.
- 큰 화면에서 정보 밀도와 공간 활용이 약하다.
- 최우선 작업이 “메뉴 진입 → 하위 메뉴 → 탭 선택”으로 남는다.

### C. 탭 관리 중심형
**장점**
- reorder / rename / delete 같은 관리 작업에는 강하다.
- 탭 리스트를 메인으로 두므로 관리형 사용자에게는 직관적일 수 있다.

**단점**
- 최우선 작업인 attach-current 흐름이 주인공이 아니다.
- “현재 pane을 어디에 붙일지”보다 “탭을 관리한다”는 인상이 먼저 온다.

---

## 4. 목표 / 비목표

### 목표
- 현재 pane을 기존 탭에 붙이는 작업을 가장 짧고 자연스럽게 만든다.
- 큰 화면에서 상태와 행동을 한 번에 이해할 수 있게 한다.
- attach / 새 탭 / 관리가 같은 화면 언어 안에서 일관되게 작동하도록 만든다.
- 탭 지속성 규칙을 UI에 자연스럽게 반영한다.

### 비목표
- tmux 바깥의 GUI를 도입하지 않는다.
- pane / window / session 모델 자체를 바꾸지 않는다.
- 저장 포맷을 새로 설계하지 않는다.
- 기존 모든 `display-menu` 경로를 즉시 제거하는 것을 목표로 하지 않는다.

## 5. 정보 구조

새 UI는 **2열 패널형**이다.

### 왼쪽 패널: 작업 목록 (항상 보임)
고정 항목:
1. 기존 탭에 붙이기
2. 새 탭에 붙이기
3. 탭 관리
4. 단축키 보기

왼쪽 패널은 “현재 어떤 모드인지”를 보여주는 내비게이션 역할을 한다. 사용자는 attach 중이든 관리 중이든 같은 위치에서 다른 모드로 바로 전환할 수 있다.

### 오른쪽 패널: 탭 리스트 (항상 보임)
항상 같은 리스트를 유지한다.

각 행은 최소한 다음 정보를 가진다.
- 탭 번호
- 탭 이름
- 상태 (`현재 탭`, `연결됨`, `비어 있음`, `연결 끊김`)
- 필요 시 짧은 연결 정보

상태 색상은 현재 메뉴 UX에서 이미 검증된 체계를 재사용한다.
- 현재 탭: 파란색 계열
- empty: 노란색 계열
- disconnected(dead): 빨간색 계열
- connected: 기본 텍스트색

---

## 6. 상호작용 설계

### 기본 상태
메뉴를 열면:
- 왼쪽 패널은 보인다.
- 포커스는 `기존 탭에 붙이기` 모드에 있다.
- 오른쪽 탭 리스트가 활성화되어 있다.
- 현재 pane의 현재 연결 상태는 상단 요약으로 보인다.

### 기본 키 동작
- `Enter`: 현재 모드의 기본 액션 실행
- 숫자: 해당 탭 즉시 선택
- `j/k` 또는 방향키: 리스트 이동
- `Esc` 또는 `q`: 닫기 또는 상위 상태 복귀
- 마우스 클릭: 리스트 항목 직접 선택

### attach-current 모드
오른쪽 탭 리스트에서 탭을 선택하면, 현재 pane이 해당 탭에 붙는다.

핵심 원칙:
- attach-current는 **추가 하위 메뉴 없이** 끝나야 한다.
- 필터나 이동 없이도 숫자와 Enter만으로 처리 가능해야 한다.

### 새 탭에 붙이기 모드
- 새 탭 이름 입력
- 입력 직후 탭 생성 + 현재 pane attach
- 성공 후 같은 패널 UI에서 생성 결과가 반영되어 보여야 한다.

### 탭 관리 모드
오른쪽 탭 리스트는 그대로 유지되고, 선택된 탭 기준으로 관리 액션을 적용한다.

세부 우선순위:
1. 순서 변경
2. 이름 변경
3. 삭제
4. dead/empty 정리

즉 “화면을 바꾸는” 방식보다 “모드를 바꾸고 같은 리스트 위에서 작업하는” 방식에 가깝다.

## 7. 상세 플로우

### 7.1 기존 탭에 붙이기
1. 메뉴 열기
2. 기본 모드가 attach-current
3. 오른쪽에서 대상 탭 선택
4. `Enter` 또는 클릭
5. attach 완료 후 상태 갱신

### 7.2 새 탭에 붙이기
1. 왼쪽에서 `새 탭에 붙이기`
2. 이름 입력
3. 탭 생성 + 현재 pane attach
4. 오른쪽 리스트 갱신

### 7.3 순서 변경
1. 왼쪽에서 `탭 관리`
2. reorder 상태 진입
3. 오른쪽에서 탭 선택
4. `위로 / 아래로 이동`
5. 리스트 즉시 갱신

### 7.4 이름 변경
1. 탭 관리 모드 진입
2. 오른쪽에서 대상 탭 선택
3. 이름 입력
4. 결과 즉시 반영

### 7.5 삭제
1. 탭 관리 모드 진입
2. 대상 탭 선택
3. 확인 후 삭제
4. 리스트 반영

### 7.6 dead/empty 정리
1. 탭 관리 모드에서 cleanup 실행
2. empty 또는 disconnected 상태 탭 제거
3. 남은 탭 리스트 즉시 반영

---

## 8. 상태 및 용어 규칙

사용자-facing 용어를 다음으로 통일한다.

- current → `현재 탭`
- connected → `연결됨`
- empty → `비어 있음`
- dead/orphaned → `연결 끊김`
- current pane에 연결된 탭이 없음 → `연결된 탭 없음`

`죽은 pane` 같은 내부 구현 느낌의 문구는 더 이상 기본 UI 용어로 쓰지 않는다.

## 9. 아키텍처 방향

### 프론트엔드
`display-menu`보다 더 큰 화면을 자연스럽게 쓰기 위해, 최종 목표는 **2열 패널형 tmux-native UI**다.

이 설계는 다음 두 구현 경로 중 하나를 허용한다.

#### 경로 A: popup 기반 커스텀 TUI
- tmux popup 안에서 2열 UI를 직접 그림
- 가장 이상적인 최종 형태
- 큰 화면 활용도와 상태/행동 동시 노출에 적합

#### 경로 B: 기존 menu 기반을 패널형에 가깝게 단계적으로 확장
- 현재 코드 위에서 점진적 개선
- 구현 부담이 낮음
- 하지만 2열 패널형의 장점을 완전히 살리기 어려움

**권고:** 최종 구현은 경로 A를 우선 검토하되, 백엔드는 지금 구조를 유지한다.

### 백엔드
현재 `common.sh`와 `pane-menu.sh`에 있는 저장/복원/attach/reorder/delete 로직은 재사용한다.

핵심 원칙:
- 프론트엔드를 바꿔도 저장 포맷은 유지
- attach / reorder / rename / delete / prune 로직은 백엔드 함수 재사용
- persistence 규칙은 회귀 테스트로 고정

---

## 10. 에러 처리 및 복구

- 저장된 탭이 없어도 UI는 정상적으로 열려야 한다.
- 현재 pane이 어느 탭에도 연결되지 않은 경우, attach-current 모드는 여전히 기본 진입 가능해야 한다.
- 대상 탭이 비어 있으면 attach 가능해야 한다.
- 대상 탭이 disconnected 상태여도 attach하면 정상 탭으로 복구되어야 한다.
- cleanup 이후 탭이 0개가 되어도 UI가 깨지지 않아야 한다.
- 현재 pane이 attach되어 있지 않은 상태에서 `현재 탭에서 해제`를 요청하면 안전하게 no-op 또는 비활성 처리한다.

---

## 11. 테스트 설계

### 단위/스크립트 테스트
- 저장된 탭 복원 시 orphan pane → empty 정규화
- attach-current가 올바른 탭 번호에 attach 수행
- reorder up/down 로직 검증
- rename/delete/prune 로직 검증
- 이전 탭 점프(`Option+\``) 회귀 테스트 유지

### 메뉴/상태 테스트
- 현재/empty/disconnected 색상 표시 유지
- 메인/attach/manage 모드 라벨 검증
- 탭 관리 후 올바른 화면/상태로 복귀

### 수동 검증
- `prefix + m`으로 진입했을 때 기본 모드가 attach-current인지 확인
- 숫자 / Enter / 마우스 클릭 모두 attach에 동작하는지 확인
- 큰 화면에서 좌우 레이아웃이 자연스럽게 읽히는지 확인
- 좁은 화면에서 최소한 깨지지 않는지 확인

---

## 12. 마이그레이션 전략

### 1단계
현재 저장/복원/attach 백엔드는 유지한다.

### 2단계
새 2열 패널형 UI를 별도 진입 경로 또는 실험 플래그로 붙인다.

### 3단계
사용감이 충분히 검증되면 기존 계층형 메뉴를 기본 경로에서 내리고, fallback 수준으로만 유지한다.

---

## 13. 리스크와 대응

### 리스크 1: 구현 복잡도 증가
큰 화면용 2열 UI는 `display-menu`보다 구현량이 크다.  
**대응:** 백엔드는 재사용하고, 화면 전환/모드 전환만 새로 설계한다.

### 리스크 2: 좁은 터미널 대응
큰 화면용 설계가 좁은 환경에서 답답할 수 있다.  
**대응:** 최소 폭 이하에서는 축약 표시 또는 fallback 메뉴를 유지한다.

### 리스크 3: 상태와 액션 과밀
한 화면에 너무 많은 정보를 넣으면 다시 복잡해질 수 있다.  
**대응:** 제목에는 핵심 상태만, 본문에는 현재 모드에서 필요한 액션만 표시한다.

---

## 14. 구현 전 최종 결정

구현은 다음 방향을 기준으로 한다.

- 2열 패널형
- 기본 진입은 attach-current
- 왼쪽 작업 목록은 항상 보임
- 오른쪽 탭 리스트는 항상 유지
- 탭 관리도 같은 리스트 위에서 처리
- persistence는 현재 규칙 유지
- orphan pane은 empty로 복원

이 설계는 현재 사용자 승인 상태이며, 다음 단계는 구현 계획 수립이다.
